# PRODUCT_SPEC

**项目**: FormZero Email Editor
**状态**: Draft
**日期**: 2026-04-09

## 背景与前提

当前仓库仍然是 FormZero 开源版的表单后端产品，而不是邮件模板编辑器。现有能力集中在：

- 表单提交接收 API
- 提交数据看板与基础分析
- CSV 导出
- SMTP 测试与通知邮件
- 登录鉴权与基础设置

这意味着你现在拥有的是一个可靠的部署底座，而不是邮件编辑器的产品内核。要对标 Beefree、Stripo、Unlayer，核心任务不是“继续加几个页面”，而是补齐一整套邮件设计文档模型、编辑器交互、渲染导出、恢复机制与质量保障链路。

---

## 1. 竞品差距分析

基于当前仓库能力，与 Beefree / Stripo / Unlayer 的共同能力集合相比，FormZero 缺失的核心功能按用户价值高低排序如下。

| 排名 | 缺失能力 | 为什么重要 |
| --- | --- | --- |
| 1 | 可视化拖拽邮件编辑画布 | 这是用户创建邮件的主入口，没有它就无法和专业编辑器处于同一赛道。 |
| 2 | 可稳定导出的响应式邮件 HTML 渲染器 | 用户最终买单的是“在 Gmail / Outlook / Apple Mail 里正常显示”，不是编辑器界面本身。 |
| 3 | 完整的区块系统与属性面板 | 没有 section / column / text / image / button / spacer / divider 等基础块，就无法高效构建真实营销邮件。 |
| 4 | 桌面 / 移动预览与所见即所得校验 | 用户必须在编辑时立即确认布局、层级和折行，否则会频繁回退重做。 |
| 5 | Undo / Redo / Autosave / Crash Recovery | 邮件编辑是高频试错工作流，没有恢复能力就不可能达到专业工具的信任标准。 |
| 6 | 模板工作区与模板生命周期管理 | 用户需要创建、复制、重命名、归档、恢复模板，而不是把邮件当一次性内容处理。 |
| 7 | 版本历史与一键回滚 | 专业编辑器必须允许“改坏了立刻回到上一个正确版本”，否则越用越不敢改。 |
| 8 | 变量 / Merge Tags / 示例数据预览 | 个性化邮件离不开占位符和假数据预览，否则导出结果无法直接投入业务。 |
| 9 | 可复用模块库与品牌样式预设 | 高频邮件生产依赖复用，而不是每次从空白画布重搭。 |
| 10 | 图片资产管理与替换流程 | 邮件生产里图片更新极其频繁，没有素材库、替换、压缩和 alt 文本就会严重拖慢效率。 |
| 11 | HTML 导入与设计文档导出 | 个人用户往往已有旧模板，没有迁移能力就很难把旧资产带进新工具。 |
| 12 | 测试发送与质量检查 | 专业工具必须在真正发送前发现缺图、空链接、按钮不可点、缺少 alt 文本等问题。 |
| 13 | 外部平台导出与集成接口 | 真正上线时通常要进入 ESP、CRM 或自动化流程，没有导出链路会卡在最后一步。 |
| 14 | 条件内容 / 动态内容 | 当模板开始服务多个受众时，条件块能显著减少模板分叉数量。 |
| 15 | 评论 / 审批 / 协作锁定 | 对单人使用不是第一优先级，但它是商业产品形成团队粘性的关键能力。 |
| 16 | 品牌治理能力 | 锁定颜色、字体、组件样式可以避免模板质量随着迭代不断走样。 |

### 竞品启示

- **Beefree** 的强项是可视化编辑、可复用内容、品牌一致性和协作工作流。
- **Stripo** 的强项是邮件模块复用、导出链路、AMP 邮件能力和针对营销场景的生产效率。
- **Unlayer** 的强项是编辑器内核、嵌入式扩展、自定义工具、动态变量与导出能力。

对你来说，真正需要先追上的不是“所有高级功能”，而是这些产品的共同底座：**稳定编辑、稳定预览、稳定导出、稳定恢复**。

## 2. 个人使用的企业级标准定义

### 定义

在个人工具语境下，“企业级”不等于多租户、RBAC、复杂审批链；它意味着：

> 单人使用时也必须像成熟商业软件一样可靠、可恢复、可预期、可扩展，任何一次编辑都不应让用户担心丢数据、导错 HTML 或卡在最后一步。

### 2.1 UX 标准

- 所有直接操作必须在 **100ms 内给出可感知反馈**；超过 **200ms** 的操作必须出现明确加载状态。
- 画布拖拽、选中、悬停高亮必须保持 **60fps 级别流畅度**，至少不能出现连续掉帧和元素错位。
- 常规模板的保存必须在后台自动完成，**2 秒内持久化**，且用户无需手动点击保存。
- 导出 HTML、切换预览、发送测试邮件都必须有清晰进度状态、成功确认和失败提示。
- 错误提示必须使用人类语言，说明 **发生了什么、影响什么、下一步怎么修复**。
- 所有破坏性操作都必须可撤销，或在不可撤销前给出明确确认。
- 预览与最终导出结果必须高度一致，不能出现“编辑器里对，邮件里错”的系统性偏差。
- 键盘输入、拖拽、选择、多选、删除等核心操作必须形成稳定心智模型，不能随机改变行为。

### 2.2 功能完整性标准

以下能力构成“个人使用也不能妥协”的核心功能清单：

- 模板 CRUD：创建、复制、重命名、归档、删除、恢复。
- 基于文档模型的可视化编辑器，而不是把 HTML 文本框包装成“编辑器”。
- 核心区块：section、column、text、heading、image、button、divider、spacer、social、html。
- 右侧属性面板：排版、间距、对齐、边框、背景、按钮状态、容器宽度。
- 桌面与移动预览，支持快速切换。
- 稳定 HTML 导出、复制、下载。
- Merge Tags 与示例数据预览。
- 图片上传、替换、裁切入口、alt 文本管理。
- Undo / Redo、Autosave、恢复未保存草稿。
- 测试发送到本人邮箱。
- 版本快照与回滚。
- 可复用模块库。

如果以上任一项缺失，就仍然更像“可演示原型”，而不是“可长期依赖的个人生产工具”。

### 2.3 代码质量标准

- **文档模型优先**：编辑器必须围绕一个版本化的 JSON 设计文档工作，UI 只是这个文档的操作界面。
- **渲染器隔离**：邮件 HTML 生成必须是独立、可测试、可复用的纯函数层，不允许散落在页面组件里拼字符串。
- **命令式编辑内核**：新增区块、移动、删除、复制、样式修改都应走统一 command/transaction 管线，便于 undo/redo 与审计。
- **严格类型约束**：核心域模型、渲染器输入输出、持久化结构不允许 `any` 漫流。
- **可迁移性**：设计文档必须带 schema version，并有明确 migration 策略。
- **扩展边界明确**：新增一个 block/tool 不应要求重写画布、渲染器或存储层。
- **测试基线明确**：P0 流程必须有端到端测试，渲染器必须有 fixture/snapshot 测试，导入导出必须有契约测试。
- **性能预算明确**：不能在一个普通模板上因为状态管理或重渲染失控而让编辑器逐渐变卡。
- **错误可观测**：保存失败、导出失败、渲染异常必须可记录、可复盘，而不是只在控制台里消失。

### 层级结论

#### P0 必须做

P0 的定义是：**做完以后，你可以独立、稳定地产出一封真实可发送的专业邮件，而不是一个演示版编辑器。**

- 模板 CRUD
- 设计文档 JSON 模型
- 可视化画布与核心区块
- 属性面板
- 桌面 / 移动预览
- 稳定 HTML 渲染导出
- Merge Tags 与示例数据
- Undo / Redo
- Autosave
- 崩溃恢复
- 测试发送

#### P1 应该做

P1 的定义是：**显著提高生产效率和长期可用性，但缺失时仍然能完成主要工作。**

- 版本历史
- 可复用模块库
- 图片资产管理
- HTML 导入
- 质量检查
- 品牌样式预设
- 外部平台导出
- Starter Templates

#### P2 可以做

P2 的定义是：**有价值，但不应该阻塞单人专业使用场景的闭环。**

- 团队协作与审批
- 角色权限
- 条件内容
- AMP
- AI 生成
- 编辑器主题和轻量增强项

## 4. 体验红线

以下 10 条属于绝不妥协的 UX 原则：

1. 任何直接用户操作都必须在 **100ms 内得到界面反馈**。
2. 任何会修改内容的操作都必须支持 **撤销**，或在执行前明确告知不可撤销。
3. 用户的工作内容 **绝不能因为刷新、崩溃、断网或误关闭页面而丢失**。
4. 所有错误都必须用 **人类语言** 描述，并附带可执行的修复建议。
5. 编辑器预览结果与最终导出 HTML 的视觉差异必须被视为 **严重缺陷**。
6. 长于 **200ms** 的任务必须显示状态，长于 **1s** 的任务必须显示进度或明确等待原因。
7. 拖拽操作的目标位置必须始终可见、可预测、可取消，不能靠猜。
8. 所有保存、导出、测试发送结果都必须有明确的成功或失败回执，不能静默结束。
9. 所有破坏性操作都必须可恢复到最近正确状态，至少能回滚到最近快照。
10. 即使只有一个人使用，系统也必须默认把“稳定、可信、可恢复”置于“炫技功能”之前。

---

## 建议的产品判断标准

如果一个版本同时满足以下 4 条，才可以被称为“专业邮件模板编辑器 MVP”：

- 可以从零创建一封响应式邮件，不需要写代码。
- 可以稳定导出可用 HTML，并成功发到自己的邮箱。
- 可以在误操作后通过撤销、恢复或回滚找回工作。
- 可以基于同一套文档模型长期迭代，而不会因为新增一个 block 就重写半个系统。
